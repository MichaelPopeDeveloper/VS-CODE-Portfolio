15h → 20m:
a SQL index story.

How a partial-text-match search on 50 years of legal-conflict data went from overnight to coffee-break — with three indexes, one filtered view, and zero vendor invoices.

MP
Michael Pope Senior AI Engineer · Las Vegas
share copy link

At Kilpatrick Townsend, attorneys run conflict checks before they take on a new matter — a partial-text-match query against decades of client names, opposing parties, and matter descriptions stored as XML blobs. The query was correct. It was also, on a bad day, taking fifteen hours.

Fifteen hours is the kind of latency that quietly reshapes a firm. Attorneys batch their conflict checks. The intake team starts running them overnight. New business waits. Eventually someone files a ticket that says "can we just buy something for this?" and a six-figure license quote lands in someone's inbox.

We didn't buy anything. We added three indexes and a filtered view. The query now finishes in about twenty minutes, and the most expensive part of the project was explaining to leadership that the answer had been hiding in the database the whole time.

The shape of the problem

Conflict data lives in IntApp, which stores most of the interesting fields as XML columns — flexible for the firm's schema, hostile to anything resembling an index. The original query did roughly this: 

conflict_search.sql
SELECT c.matter_id, c.opened_at, c.client_name
  FROM conflicts c
  WHERE c.payload.value('(/m/parties/party/name)[1]', 'nvarchar(400)')
        LIKE '%' + @term + '%'
     OR c.payload.value('(/m/parties/party/aka)[1]', 'nvarchar(400)')
        LIKE '%' + @term + '%'
     OR c.description LIKE '%' + @term + '%'
-- 50 years × ~12M rows × XML.value() per row × LIKE '%term%'
-- = a query plan that reads as cosmic horror

Every .value() call shreds XML at runtime. LIKE '%term%' defeats any normal B-tree. The optimizer's only move is a full scan, repeated three times in disjunction, on a table that doesn't fit in memory. Of course it took fifteen hours.

What we changed

The fix had three parts, and each one is unglamorous in isolation:

1. Promote the searchable fields out of XML

Computed columns + PERSISTED. Pull the three fields the query actually touches into real, indexable storage. SQL Server materializes them on insert, and now they look like ordinary nvarchar to the planner. 

001_promote_fields.sql
ALTER TABLE conflicts ADD
  party_name AS payload.value('(/m/parties/party/name)[1]', 'nvarchar(400)') PERSISTED,
  party_aka  AS payload.value('(/m/parties/party/aka)[1]',  'nvarchar(400)') PERSISTED;

2. Full-text indexes, not B-trees

LIKE '%term%' can't use a B-tree, but a full-text index can answer CONTAINS in milliseconds. We rewrote the predicate. The attorneys did not care which keyword we used; they cared that the screen stopped taking overnight. 

002_fulltext.sql
CREATE FULLTEXT INDEX ON conflicts (party_name, party_aka, description)
  KEY INDEX pk_conflicts
  WITH change_tracking AUTO;

-- query becomes:
SELECT matter_id, opened_at, client_name
  FROM conflicts
  WHERE CONTAINS((party_name, party_aka, description), '@term');

3. A filtered view for the hot path

90% of conflict checks only care about open or recently-closed matters. A filtered indexed view over the last 24 months gave the planner a much smaller working set to scan in the rare case CONTAINS couldn't help — like very short or fragmentary search terms.

Indexes are not magic. They are a contract: you trade write throughput and disk for read latency. On a system that ingests 800 conflicts a day and reads them ten million times, that contract is laughably one-sided. — a comment I left on the PR

What I learned (again)

  • XML columns are write-optimized. If you read them in a hot path, you are paying for flexibility you may not need. Promote the fields. The schema migration is one afternoon.
  • Full-text exists. SQL Server's full-text engine is older than I am and still the right answer for "find all rows where any of these three columns contains a substring."
  • The first index is rarely the last. We added two more after watching a week of production query plans. The filtered view came from the third sprint, not the first.
  • Talk to the people who run the query. The intake team had been quietly running these at 11pm for two years. Nobody on the engineering side knew until I asked.

The numbers, finally

After deploy:

The vendor proposal got quietly closed. The intake team stopped running batches at midnight. And six months later, the same query is still finishing in under half an hour with another year of data added on top.

Not every problem needs a vector store. Sometimes it just needs the index it should have had in the first place.
