the problem

Attorneys were burning hours on the same five questions.

Kilpatrick Townsend handles complex IP, trademark, and litigation work. Every matter starts with the same research patterns: find the relevant precedent, summarize this brief, compare these two contracts, locate this clause across our archive. Senior associates were doing it well. Junior associates were doing it slowly. Everyone was doing it manually.

Out-of-the-box ChatGPT was a non-starter — client data couldn't leave the firm's tenant, and attorneys needed answers grounded in their documents and precedents, not the open internet. Off-the-shelf legal AI vendors quoted multi-million dollar contracts. Leadership asked: can we build it ourselves, faster, and have it actually work?

the routing layer

Pick the cheapest model that gets it right.

The hardest design decision wasn't picking a model — it was picking a strategy for picking models. Different queries have wildly different cost/quality profiles. A docket lookup is trivial; a multi-document summarization with citation alignment is not. We built a small classifier that scores each query along three axes (length, retrieval count, reasoning depth) and dispatches to the right tier.

RouterPolicy.cs
public class RouterPolicy
{
    public LlmTarget Pick(QueryFeatures q)
    {
        // cheap path: short, single-doc, low reasoning depth
        if (q.Tokens < 1200 && q.Reasoning < 2 && q.RetrievalCount <= 3)
            return LlmTarget.KtLlama;

        // everyday: medium scope, no citation alignment
        if (q.Tokens < 8000 && !q.RequiresCitations)
            return LlmTarget.GeminiFlash;

        // long-context: multi-doc summaries, contracts
        if (q.Tokens > 32000 || q.DocCount > 5)
            return LlmTarget.ClaudeSonnet;

        // expensive path: complex reasoning, mission-critical
        return LlmTarget.Gpt4o;
    }
}

The result: 58% of queries route to the cheap tier, 24% to mid, and only 18% hit premium models. We measured cost-per-answer against a single-model baseline (GPT-4 only) and came in 63% lower at the same or better quality, judged on a weekly sample of 50 queries reviewed by senior partners.

how it shipped

Eight months, three false starts.

Most "how we built it" posts skip to the architecture diagram. The actual story has a lot of backtracking. Here's the real timeline.

May 2024
Discovery + first prototype
Two weeks shadowing attorneys. Built a single-model GPT-4 demo over a sample of 5,000 documents to prove the concept and get budget approval.
Jun — Jul 2024
Ingestion pipeline
Go scraper for Harvard / Cornell legal databases via BrightData. Python chunker. First version of the vector store with 200K chunks.
Aug 2024
First false start: pure RAG wasn't enough
Citations were hallucinating. Long contract summaries were truncating. Retrieval recall was 71% — not good enough for legal work where one missed precedent is malpractice.
Sep — Oct 2024
Hybrid retrieval + Semantic Kernel orchestration
Switched to BM25 + dense hybrid search. Brought in Semantic Kernel for the agent loop — function calling for "fetch precedent", "compare clauses", "verify citation". Recall jumped to 94%.
Nov 2024
Multi-LLM router goes live
Started with two models (GPT-4 + Gemini). Added Claude in week 2 after long-context tests. Cost dropped 40% week-over-week.
Jan 2025
KT-Llama fine-tune
Trained a 70B Llama variant on the firm's own (anonymized) precedent corpus. Hosted on-prem for sensitive matters. Now handles 24% of queries at near-zero marginal cost.
Mar 2025
Rolled out to all 650 attorneys
Phased rollout: IP team first, then trademark, then full firm. Tracked thumbs feedback per response and fed it back into the fine-tune queue weekly.
Now
Agent workflows + matter-specific memory
Building per-matter context windows so the agent remembers what it has already retrieved within an active case. Early tests show another ~30% time savings on long-running matters.

what I learned

Five lessons that generalize.

  1. 01Routing beats picking. The "best LLM" question is a trap. The actually-useful question is which model for which sub-task, and you only learn that in production.
  2. 02Verification is the product. Lawyers don't trust answers they can't trace. We spent more engineering hours on citation alignment and source attribution than on the model layer itself.
  3. 03Domain fine-tunes pay rent. A 70B model trained on the firm's own corpus handled a quarter of all traffic at a tenth of the cost — but only after retrieval was good enough to feed it clean context.
  4. 04Shadow the user before you architect. Two weeks watching paralegals run conflict checks reshaped the whole UI. You can't pull this from a Jira ticket.
  5. 05Ship the boring version first. The single-model demo got us budget. Everything good came after — but only because the unglamorous v1 existed and people were already using it.
